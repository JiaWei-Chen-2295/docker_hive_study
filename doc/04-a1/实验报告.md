## 4 安装配置过程中遇到的问题与解决

### 4.1 容器启动顺序问题

**问题描述：** 初次执行 `docker compose up` 时，Hive 容器启动失败，日志显示无法连接 MySQL 元数据库。

**原因分析：** Docker Compose 默认同时启动所有容器，但 MySQL 初始化需要一定时间（约 15-20 秒），Hive 在 MySQL 尚未就绪时尝试连接导致失败。

**解决方法：** 
- 方案一：在 `docker-compose.yml` 中为 Hive 服务添加 `depends_on` 依赖配置，并配合健康检查
- 方案二：先启动 MySQL 和 Hadoop，等待其完全启动后再启动 Hive
- 实际采用方案二，分步启动服务并观察日志确认就绪状态

### 4.2 HDFS 目录权限问题

**问题描述：** 初始化 HDFS 目录时，出现 `Permission denied` 错误，无法创建 `/user/hive/warehouse` 目录。

**原因分析：** Hadoop 默认开启权限检查，当前执行用户没有 HDFS 根目录的写权限。

**解决方法：** 
- 先使用 `hadoop fs -chmod -R 777 /tmp` 放开临时目录权限
- 使用 `hadoop fs -chown -R hive:hive /user/hive` 将 Hive 目录所有权赋给 hive 用户
- 也可以在 `hdfs-site.xml` 中设置 `dfs.permissions.enabled=false` 关闭权限检查（不推荐用于生产环境）

### 4.3 HiveServer2 启动时间较长

**问题描述：** 执行 `docker logs -f hive-server` 后，等待较长时间才看到 `Starting HiveServer2` 的输出。

**原因分析：** HiveServer2 启动时需要完成以下初始化步骤：
1. 连接 MySQL 元数据库并验证元数据 schema
2. 初始化会话管理器
3. 加载所有内置函数
4. 启动 Thrift 服务监听 10000 端口

整个过程通常需要 1-2 分钟，属于正常现象。

**解决方法：** 耐心等待日志输出，不要过早尝试连接。可以通过 `docker logs hive-server 2>&1 | grep "Starting HiveServer2"` 来快速确认启动状态。

### 4.4 Beeline 连接被拒绝

**问题描述：** 使用 Beeline 连接时出现 `Connection refused` 错误。

**原因分析：** HiveServer2 尚未完全启动完成，Thrift 服务还未开始监听 10000 端口。

**解决方法：** 
- 确认 HiveServer2 日志中已出现 `Starting HiveServer2` 字样
- 使用 `docker exec hive-server netstat -tlnp | grep 10000` 确认端口已监听
- 等待 30 秒后重试连接

## 5 业务需求分析与实现

### 5.1 需求背景

作为游戏公司用户部门员工，需要分析用户游戏时长数据。数据包含以下字段：

| 字段 | 说明 |
|------|------|
| user_id | 用户 ID |
| p_date | 日期 |
| gender | 性别 |
| age | 年龄 |
| duration | 当日在线时长（小时） |

数据来源为 1-3 月某 3 天的用户数据文件（001.txt、002.txt、003.txt），另有 city.txt 包含用户城市信息。

### 5.2 分区策略选择

**选择分区字段：** `p_date`（日期）

**选择理由：**
1. 数据天然按日期组织，每个文件对应一个日期
2. 按日期分区后，查询特定日期数据时可避免全表扫描，提高查询效率
3. 便于按时间范围进行统计分析，符合业务查询习惯
4. 便于后续按日期增量导入新数据

### 5.3 数据导入实现

#### 5.4.3 为法务部门创建临时表

**需求：** 筛选年龄小于 15 且曾经有当日在线时长大于 2 小时的用户。

**实现：**

```sql
-- 创建临时表存储高时长用户
DROP TABLE IF EXISTS high_duration_users;
CREATE TABLE high_duration_users AS 
SELECT DISTINCT user_id 
FROM game_user 
WHERE duration > 2;

-- 创建法务部门用户表
DROP TABLE IF EXISTS legal_dept_users;
CREATE TABLE IF NOT EXISTS legal_dept_users (
    user_id STRING,
    gender STRING,
    age INT
);

-- 设置 join 参数避免小表 join 问题
SET hive.auto.convert.join=false;

-- 插入符合条件的用户数据
INSERT INTO legal_dept_users 
SELECT DISTINCT a.user_id, a.gender, a.age 
FROM game_user a 
JOIN high_duration_users b ON a.user_id = b.user_id 
WHERE a.age < 15;

-- 查看结果
SELECT * FROM legal_dept_users;
```

**选择这种表的原因：**

1. **使用中间表（high_duration_users）的原因：** 将"在线时长大于 2 小时"的用户先提取为独立表，避免在后续 JOIN 中重复扫描大表，提高查询效率
2. **使用普通表而非临时表的原因：** 法务部门可能需要多次查询这批用户数据，普通表可以持久化存储，避免重复计算
3. **使用 DISTINCT 去重：** 同一用户可能在多个日期满足条件，需要去重保证结果唯一
4. **设置 `hive.auto.convert.join=false`：** 防止 Hive 自动将 JOIN 转换为 MapJoin 时因表大小判断不准确导致的问题

#### 5.4.4 用户画像分析

**性别比分析：**

```sql
SELECT 
    COUNT(DISTINCT user_id) AS total_users,
    SUM(CASE WHEN gender = 'm' THEN 1 ELSE 0 END) AS male_count,
    SUM(CASE WHEN gender = 'f' THEN 1 ELSE 0 END) AS female_count,
    SUM(CASE WHEN gender = 'm' THEN 1 ELSE 0 END) / SUM(CASE WHEN gender = 'f' THEN 1 ELSE 0 END) AS gender_ratio
FROM (
    SELECT user_id, gender 
    FROM game_user 
    GROUP BY user_id, gender
) t;
```

**年龄分布分析：**

```sql
-- 15岁以下
SELECT 
    COUNT(DISTINCT user_id) AS under_15_count
FROM (
    SELECT user_id, age 
    FROM game_user 
    GROUP BY user_id, age
) t
WHERE age < 15;

-- 15-20岁

    COUNT(DISTINCT user_id) AS age_20_25_count

    SELECT user_id, age

    COUNT(DISTINCT user_id) AS above_25_count
```

#### 5.4.5 广告投放筛选（city 表关联查询）

**需求：** 筛选位置在北京、上海、广州且 15 岁以上的用户。

**实现思路：**

```sql
-- 创建 city 外部表
CREATE EXTERNAL TABLE IF NOT EXISTS city (
    user_id STRING,
    city STRING
)
ROW FORMAT DELIMITED 
FIELDS TERMINATED BY '\t' 
STORED AS TEXTFILE 
LOCATION 'hdfs://hadoop-namenode:8020/user/hive/warehouse/game4202.db/city';

-- 筛选目标用户
SELECT DISTINCT a.user_id, a.gender, a.age, b.city
FROM game_user a
JOIN city b ON a.user_id = b.user_id
WHERE b.city IN ('beijing', 'shanghai', 'guangzhou')
  AND a.age > 15;
```

**说明：** 使用 `IN` 子句匹配三个目标城市，结合年龄条件进行过滤，`DISTINCT` 确保用户不重复。

#### 5.4.6 附加题：在线时长持续降低的用户

**需求：** 查询当日在线时长持续降低且 15 岁以上的用户。

**实现思路：**

```sql
-- 方法：使用窗口函数 LAG 比较相邻日期的时长
WITH user_duration AS (
    SELECT 
        user_id,
        p_date,
        duration,
        LAG(duration) OVER (PARTITION BY user_id ORDER BY p_date) AS prev_duration
    FROM game_user
),
declining_users AS (
    SELECT DISTINCT user_id
    FROM user_duration
    WHERE prev_duration IS NOT NULL
      AND duration < prev_duration
    GROUP BY user_id
    HAVING COUNT(*) = 2  -- 连续两次都在降低（3个时间点需要2次比较）
)
SELECT a.user_id, a.gender, a.age
FROM game_user a
JOIN declining_users b ON a.user_id = b.user_id
WHERE a.age > 15
GROUP BY a.user_id, a.gender, a.age;
```

**说明：** 
1. 使用 `LAG()` 窗口函数获取每个用户上一次记录的时长
2. 比较当前时长与上次时长，筛选出降低的记录
3. 对于 3 个时间点的数据，连续降低意味着 2 次比较都满足 `duration < prev_duration`
4. 最后关联用户基本信息并筛选年龄大于 15 的用户

## 7 实验过程中遇到的问题与解决

### 7.1 数据导入时分区字段处理

**问题描述：** 初次尝试直接向分区表插入数据时，没有正确使用 `PARTITION` 子句，导致数据未能正确分区。

**解决方法：** 使用 `INSERT OVERWRITE TABLE table_name PARTITION (partition_col=value)` 语法，确保每个分区的数据独立导入。

### 7.2 JOIN 查询性能问题

**问题描述：** 在法务部门用户筛选时，直接使用子查询进行 JOIN，Hive 默认尝试使用 MapJoin 优化，但因表大小判断不准确导致执行缓慢。

**解决方法：** 设置 `SET hive.auto.convert.join=false;` 关闭自动 MapJoin 转换，使用常规的 Reduce-Side JOIN，执行更稳定。

### 7.3 窗口函数兼容性问题

**问题描述：** Hive 4.0.0 对窗口函数的支持较之前版本有所改进，但某些复杂窗口函数仍可能出现语法错误。

**解决方法：** 
- 使用 `LAG()` 函数时确保 Hive 版本支持（Hive 4.0.0 已支持）
- 窗口函数中的 `ORDER BY` 必须明确指定排序字段
- 复杂查询使用 CTE（`WITH` 子句）分步实现，提高可读性和调试效率

### 7.4 数据类型转换问题

**问题描述：** 原始数据文件中 age 字段为整数，duration 为浮点数，在建表时需要正确指定数据类型。

**解决方法：** 
- age 使用 `INT` 类型
- duration 使用 `DOUBLE` 类型以保留小数精度
- user_id 使用 `STRING` 类型（因为 ID 可能包含前导零，如 "001"）

### 7.5 Beeline 输出格式问题

**问题描述：** Beeline 默认输出格式不够直观，列对齐不整齐，不利于查看结果。

**解决方法：** 
- 使用 `!set outputformat vertical` 切换为垂直输出格式
- 或使用 `!set outputformat table` 使用表格格式
- 对于少量数据的查询结果，默认格式已足够

4. **复杂查询：** 使用 JOIN、子查询、窗口函数等完成了法务部门用户筛选、用户画像分析、广告投放筛选等复杂业务需求

通过本次实验，深入理解了 Hive 依赖 HDFS 存储数据、依赖 MySQL 存储元数据的整体架构，掌握了分区表的设计与使用，为后续数据仓库开发奠定了坚实基础。
